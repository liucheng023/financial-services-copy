## Problem Statement

Anthropic 的 financial-services 项目提供了 10 个金融 Agent、7 个垂直技能包、50+ 个技能和 11 个 MCP 数据连接器，但这些能力目前只能通过 Claude Cowork（交互式）或 Managed Agent API（无头）两种方式使用。我们需要构建第三个交付渠道——Web 可视化后台，最终目标是打造金融版的 Omniwork.ai：一个面向金融工作场景的 Agent OS。

## What Makes This Cool

一个银行分析师打开浏览器，选择 Pitch Agent，输入"帮我做 AAPL 的 comps 分析"——Agent 自动从 Daloopa/FactSet 拉数据、跑分析、生成 Excel。切换到 GL Reconciler，丢进去总账文件，它自动找差异、追根因。这不是聊天，是干活。

## The Real Moat

Omniwork.ai 证明了"Agent OS"模式在创作领域可行。但 Omniwork 的专家是内容创作领域的。金融领域的 Agent 需要：
- 真正的机构级专业知识（不是泛泛而谈）
- 真实的金融数据接入（MCP 连接器）
- 合规护栏（每个数字必须有来源，不能编造）

我们站在 40,000 行机构级金融专业知识的提示词之上。平台是交付工具，内容才是护城河。

## Constraints

1. **模型无关**：不能 only for Claude，必须支持 OpenAI 兼容格式（GLM-5 优先）
2. **MCP 适配已解决**：使用 mcp-use 的 OpenAIMCPAdapter，不自建
3. **内容原样复用**：39,893 行 markdown 不重写，通过适配层桥接
4. **Supabase**：数据库 + 对象存储 + 聊天记录
5. **框架优先**：先搭完整框架，2-3 个 Agent 可运行，再逐步完善

## Premises (All Agreed)

1. Agent prompts 是模型无关的领域知识——只需适配工具调用格式
2. MCP → Function Calling 是已解决的问题（mcp-use 等）
3. 内容是护城河，基础设施是交付工具
4. 框架优先，深度第二——先展示所有 Agent，2-3 个可运行

## Approaches Considered

### Approach A: Pipeline First
- 最小可行：Agent 列表 + 单 Agent 对话 + 模型配置
- Effort: M | Risk: Low
- 缺点：只是聊天 UI，不是 Agent OS

### Approach B: Full Agent OS
- 完整 Omniwork 对标
- Effort: XL | Risk: High
- 缺点：范围太大，可能什么都交付不了

### Approach C: Incremental OS (CHOSEN)
- Phase 1: 基础平台（Agent + 对话 + MCP + 模型配置）
- Phase 2: 多 Agent 协作 + 技能沉淀
- Phase 3: 记忆系统 + 周期自动化
- 每个阶段独立交付价值

## Recommended Approach: Incremental OS

### Phase 1: FinAgentOS Foundation

**核心页面：**

| 页面 | 功能 | 优先级 |
|---|---|---|
| 对话页 | 主聊天窗口，默认通用模式，可切换 Agent | P0 |
| Agent 列表/详情 | 10 个 Agent，展示角色/工作流/技能/MCP/护栏 | P0 |
| 技能包列表/详情 | 7 个 Vertical，展示 Skills 和 MCP | P0 |
| MCP 管理 | 11 个 MCP 展示 + 新增 | P0 |
| 系统配置 | OpenAI 兼容模型配置 | P0 |
| 聊天记录 | 历史对话，按 Agent 分类 | P1 |

**Sidebar 结构：**
```
💬 默认对话
────────────
👥 专家 (Agents)
📦 技能包 (Verticals)
🔌 MCP 连接器
────────────
📋 聊天记录
⚙️ 系统配置
```

**Agent 详情页展示：**
- 角色描述（from agents/<slug>.md）
- 工作流步骤（Workflow section）
- 产出物（What you produce）
- 使用的技能列表（Skills this agent uses）
- 连接的 MCP（from frontmatter tools:）
- 安全护栏（Guardrails）

**对话交互：**
1. 默认进入通用对话模式（无 Agent）
2. 选择 Agent → 加载 system prompt + skills 知识 + MCP 连接
3. 对话中切换 Agent → 确认后清空上下文重新开始
4. Agent 的 system prompt 作为对话的 system message
5. Skills 知识注入到 system prompt 末尾或作为 context

**技术架构：**
```
┌─────────────┐     ┌──────────────────┐     ┌──────────────┐
│  Web 前端    │────→│  后端 API        │────→│  LLM (GLM-5) │
│  (React)     │     │  (FastAPI)       │     └──────────────┘
└─────────────┘     │                  │          ↑ ↓
                     │  1. 加载 Agent   │     function call
                     │     system prompt │          ↑ ↓
                     │  2. 注入 Skills   │     ┌──────────────┐
                     │     知识内容       │────→│  MCP Servers  │
                     │  3. MCP 适配层    │     │  (Daloopa...) │
                     │     (mcp-use)     │     └──────────────┘
                     │  4. 对话循环管理   │
                     └──────────────────┘
                              │
                     ┌────────┴─────────┐
                     │  Supabase        │
                     │  - agents/skills │
                     │  - chat_sessions │
                     │  - chat_messages │
                     │  - model_configs  │
                     │  - mcp_servers    │
                     └──────────────────┘
```

**MCP 适配层（核心创新点）：**
- 使用 mcp-use 的 OpenAIMCPAdapter
- 从项目 markdown 文件解析 Agent 配置，导入 Supabase
- 运行时：加载 Agent prompt → 注入 Skills → 初始化 MCP 连接 → 转为 OpenAI function schema → 对话循环

**数据模型（Supabase）：**
```sql
-- Agents
agents: id, name, slug, description, system_prompt, workflow, 
       guardrails, outputs, created_at, updated_at

-- Verticals (技能包)
verticals: id, name, slug, description, created_at

-- Skills
skills: id, name, slug, description, vertical_id, content, created_at

-- MCP Servers
mcp_servers: id, name, slug, url, api_key_encrypted, description, created_at

-- 关联表
agent_skills: agent_id, skill_id
agent_mcps: agent_id, mcp_id
vertical_skills: vertical_id, skill_id
vertical_mcps: vertical_id, mcp_id

-- 对话
chat_sessions: id, agent_id, model_config_id, title, created_at
chat_messages: id, session_id, role, content, tool_calls, tool_results, created_at

-- 模型配置
model_configs: id, name, base_url, api_key_encrypted, model_name, created_at
```

**Agent prompt 适配策略：**
- system prompt：原样使用，从 markdown 解析
- Skills 知识：作为 system prompt 的附加上下文注入
- 工具声明：`tools: Read, Write, Edit, mcp__capiq__*` → 由 mcp-use 自动转换为 function calling schema
- 工作流引用：`` `comps-analysis` `` → 技能内容注入到上下文

### Phase 2: Multi-Agent Collaboration

- 多 Agent 任务分解：复杂任务自动拆解为子任务
- Agent 间 handoff：一个 Agent 完成子任务后交接给下一个
- 技能沉淀：完成任务后一键封装为可复用 Skill
- 对标 Omniwork 的"多专家协同作业"

### Phase 3: Memory + Automation

- 用户画像记忆：记住用户角色和偏好
- 偏好记忆：记住创作/分析风格
- 项目记忆：记住当前项目上下文
- 周期性自动化：定时任务（每日追踪、每周报告）
- 对标 Omniwork 的"三层记忆系统"和"周期性自动任务"

## Open Questions

1. **默认 Agent**：默认对话用哪个 Agent？还是纯通用模式（无 system prompt）？
2. **Skills 注入策略**：全部注入 system prompt（token 开销大）还是按需检索（需要 RAG）？
3. **文件产出**：Agent 生成 Excel/PPTX 时，前端如何展示？下载链接？在线预览？
4. **认证**：是否需要用户认证？还是内部工具先不做？
5. **MCP API Key 管理**：部分 MCP 需要 API key（Daloopa、FactSet），如何安全存储？

## Success Criteria

- [ ] 所有 10 个 Agent 在页面上可见，详情页展示完整
- [ ] 至少 2-3 个 Agent 可以实际对话
- [ ] MCP 连接器可配置，至少 1 个 MCP 能实际调用
- [ ] GLM-5 模型配置可用，对话正常

- [ ] 聊天记录可保存和回看

## What I noticed about how you think

- 你说"先搭架子，某个 Agent 能跑就行，再逐步完善"——这是系统思维，不是功能思维。你在构建一个可扩展的框架，不是一个功能列表。
- 你主动纠正了"专家(skill)"的概念映射——Agent ≠ Skill。这种精确性会影响产品设计，因为 UI 必须反映正确的概念层级。
- 你明确说"不是 only for Claude"——这不是技术偏好，是产品定位。你在构建一个开放平台，不是一个 Claude 的前端。
- 你用 Omniwork.ai 做对标——说明你在用"已验证的模式 + 新的领域"来降低风险，这是务实的做法。
